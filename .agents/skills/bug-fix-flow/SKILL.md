# 开发与修复主流程（bug 修复 / 需求开发通用）

一条主线，每步**做完、确认干净再进下一步**：

1. **通读代码、定位问题** —— 先把相关代码读通，定位到真正的问题点/改动点（方法见 `debugging` 技能），别急着写；能复现的 bug 先写一个失败用例。
2. **先写方案 / 修复文档** —— 动代码前先把方案写成文档。**问题复杂就拆成多个文档**，每个聚焦一块，让每次只在最合适的上下文强度内工作，避免在一个超大上下文里顾此失彼。
   - 推荐**三件套**（放 `tmp_docs/<feature>/`，小改可合一篇）：
     - `requirements.md` —— 问题/目标 + **验收标准**。验收建议用 **EARS 记法**让它可测：`当 <条件/事件>，系统应当 <可观察的行为>`（如「当用户连续登录失败 5 次，系统应当锁定账号 15 分钟」）。
     - `design.md` —— 根因、改动方案、影响面、涉及的文件/接口、风险与回滚。
     - `tasks.md` —— 拆成带状态的清单：`- [ ] T1 …（依赖: 无；验收: 满足 requirements R2）`，每个任务**回链需求**、标依赖与状态。
3. **反复审方案** —— 对着三件套反复检查，直到找不出漏洞和遗漏（需求是否可测、design 是否覆盖边界、tasks 是否回链需求），再进入编码。
4. **写代码** —— 按审定的方案实现，最小改动、风格随上下文、遵循编码质量约束。
5. **反复审代码** —— 写完反复自检直到确认无误（正确性/边界/安全/回归），并清掉调试日志和临时代码。
6. **测试** —— 进本地或远程测试环境（**本地 Docker 优先**，详见 `env-testing` 技能），**把所有能测的都测完**：
   - 暂时测不了的，先**想办法**（造数据 / mock / 补环境）。
   - 想办法的代价或复杂度过高，**交给用户决定是否继续**。
   - 确实测不了的，**如实告知用户哪部分没覆盖、为什么**，绝不假装测过。
7. **交付部署说明** —— 全部确认无误后，**告诉用户这个项目要怎么部署到生产环境才能生效**（构建/镜像/数据库迁移/发布/重启等，按项目实际步骤），并记录本次根因与修法便于回溯。

## AI 生成文档的归置
- 本流程产出的方案/修复文档，以及其它 AI 生成的文档，统一放到**项目根目录的 `tmp_docs/`** 下。
- **同一个问题/需求的文档放进同一个子文件夹**（如 `tmp_docs/登录超时排查/` 内放三件套），别散落各处。
- **文件名要一眼看懂是干嘛的**（`requirements.md`/`design.md`/`tasks.md`，或 `01-根因分析.md` 等）。
- **完成后归档**：feature 收尾后把它的文件夹移到 `tmp_docs/_archive/<feature>/`，留作下次上下文与可追溯记录，而不是删掉。
- `tmp_docs/` 属临时产物，**建议加入 `.gitignore`**，不随代码提交（除非用户要求保留）。
