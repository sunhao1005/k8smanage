# 调试与排障（功能型 bug 的硬规则）

1. **先让真实失败现形再动手** —— 暴露被吞的异常、别在输出/渲染层 filter 掉 error 消息、看运行时日志，拿到那条真实报错，而不是靠症状反复空转。
2. **端到端追到最底层** —— 「连得上 / 收得到」≠「功能能用」。追问执行单元在它**真实运行的环境**里到底能不能做这件事，一路追进最底层（沙箱/子进程/代理/驱动）。
3. **治根因不治症状** —— 警惕「X 正常 → 复用 X 逻辑就行」的框定，先确认 X 是否真走了出问题的那条路径；别用「X 没问题」排除真正有差异的那一面。
4. **分清「功能没实现」与「逻辑写错了」** —— 缺子系统/集成靠读现有代码找不到，只能靠真实报错暴露缺口。
5. **别把「在改东西」当「在排障」** —— 致因未定位前那些容易又可见的微调，都是「路灯下找钥匙」。
6. **不确定根因先定位再改** —— 加日志、打印、诊断脚本，不靠猜测直接改代码。

> 定位到根因后，回到 `bug-fix-flow` 走「写方案→编码→测试→部署说明」的主流程。
